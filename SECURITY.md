# Security Policy

**Research Use Only. Not for clinical use.**

This document covers:
- reporting security vulnerabilities in the codebase, and
- routing medical safety incidents (adverse events) that may arise from misuse or experimentation.

## Reporting a Security Vulnerability

If you discover a security vulnerability:
- Do not open a public issue with exploit details.
- Prepare a report containing:
  - affected component and version/commit
  - reproduction steps
  - impact assessment
  - suggested mitigation (if known)

Preferred reporting routes:
- If hosted on GitHub and vulnerability reporting is enabled, use the repository “Report a vulnerability” feature.
- Otherwise, open an issue titled “Security report request” **without** sensitive details; maintainers will respond with a secure channel.

**Expected acknowledgement:** within 7 days

## Medical Safety Incidents (Adverse Events)

If you believe the software contributed to an adverse event (e.g., seizure, injury, significant distress):
- stop using the software immediately
- seek appropriate medical assistance
- report the event to the maintainers using the channel below

Reporting route:
- Open an issue titled “Safety incident report” and include the information below. Do not include personal identifiers or patient data.

**Report format:** include a short description, environment (device model, OS), steps taken, and timing. Do not include personal identifiers or patient data.

## Coordinated Disclosure

Maintainers aim to:
- acknowledge reports promptly
- assess severity and scope
- publish mitigations and release notes

## Scope Notes

This repository is not intended to process patient data. Reports containing PHI/PII will be deleted or redacted.
